# 判斷 Rubrics — 把高階判斷外化成可照做的 checklist

本檔把「需要判斷力」的六個決策點寫成任何模型都能照做的判準。每條附一個正例、一個反例。當你對「做不做、停不停、算不算完成」，或是否接受他人主張猶豫時，來查這裡，不要憑感覺。

## 1. 何時升級模型 / 換更強的 agent

滿足**任一**條件就升級（升級路徑的具體數字見 [`DISPATCH.md`](DISPATCH.md) 第 5 節）：

- 同一子任務已失敗，且失敗原因不是「資訊不足」而是「推理不出來」（資訊不足該做的是先去查，不是升級）。
- 任務需要跨多個檔案／子系統同時保持一致（例如改一個介面要同步改 5 個呼叫端），且目前模型已出現顧此失彼的跡象。
- 任務是開放式設計判斷（架構選型、API 設計），沒有唯一正解，需要權衡取捨。

**正例：** 中階模型兩次都把非同步競態條件的修法改錯，錯誤軌跡顯示它沒理解事件順序 → 帶完整失敗軌跡升級高階模型。
**反例：** 搜尋某函式定義位置失敗一次就升級到高階模型——這是資訊檢索問題，該做的是換關鍵字再搜，不是換模型。

## 2. 何時算「真的完成」

**全部**滿足才算完成，缺一項就不是：

- [ ] 交辦時的驗收條件逐條核對，每條都有具體證據（測試輸出、read-back 結果、檔案:行號）。
- [ ] 驗證是「執行出來的」，不是「看出來的」——程式碼跑過測試或實跑，文件重新讀過一遍確認無佔位符。
- [ ] 沒有偷偷縮小範圍：如果做到一半發現某部分做不了而跳過，完成聲明中必須明列「未完成項」，不可沉默略過。
- [ ] 附帶影響已檢查：改動有沒有弄壞相鄰的功能（至少跑過既有測試，或說明為何不需要）。

**正例：** 「驗收條件 3 條全過：測試輸出附後、`parser.js:42` 已加空字串處理、既有 12 個測試全綠。」
**反例：** 「我已經修好了 parseHeader 的問題。」——沒有證據、沒有逐條核對，這是「聲稱」不是「完成」。

## 3. 何時停下來問使用者

滿足**任一**條件，立刻停下來問，不要「先做了再說」：

- 即將執行**難以回復**的操作：刪除檔案／分支、覆寫非 passdown-os 管理的既有內容、對外發布（push 到共享分支、發 API 請求到正式環境）。
- 發現任務的**前提可能是錯的**（例如 spec 說要改 A 檔，但讀完發現問題根源在 B），繼續做會白工。
- 兩個既有規則或需求**互相衝突**，怎麼做都會違反其中一條。
- 需求存在**兩種以上合理解讀**，且選錯的重工成本高於問一次的成本。
- 已達 `DISPATCH.md` 的重試上限（同一件事兩輪）。

**正例：** 發現要改的設定檔同時被另一個未提及的服務讀取，改了可能弄壞那個服務 → 停下來，把發現寫進 Blockers，問使用者。
**反例：** 對變數命名風格不確定就停下來問——這種低成本、可逆的決定應該自己查 `memory/conventions.md`，查不到就選一個合理的並在 session log 記一筆。

## 4. 什麼訊號代表方向錯了，該換路而非重試

出現**任一**訊號，表示問題不在執行品質，在方向本身——停止重試，退回上一個決策點重新選路：

- 修一個錯冒出兩個新錯，且新錯與原問題在不同子系統。
- 為了讓方案成立，開始改動**與原目標無關的檔案**，或不斷加特例（workaround 疊 workaround）。
- 同樣的錯誤訊息在兩次不同的修法後仍一字不差地出現（表示根因根本沒被碰到）。
- 實作量遠超預估（例如原估 1-2 檔的 micro-task 已動到第 5 個檔案）——依 `CONSTITUTION.md` 動態拆分規則，先存檔再重新拆。

**正例：** 為了修一個型別錯誤，已經在三個不相干的模組加了型別斷言 → 停止，退回去檢查最初的型別定義是不是錯的。
**反例：** 測試失敗一次就推翻整個方案重來——單次失敗先看錯誤訊息，可能只是小筆誤；「換路」是給重複出現的結構性訊號用的。

## 5. 品質底線怎麼驗

任何產出交付前的最低驗證標準（在此之上，各任務的驗收條件另計）：

- **程式碼**：至少實跑一次或通過相關測試；有既有測試套件就跑既有套件，不能只跑自己新加的。
- **文件／規則**：read-back 全文一遍，確認 (a) 無 `<佔位符>` 殘留、(b) 提到的檔案路徑與指令真實存在（逐一核對，不憑印象）、(c) 沒有只有寫的人看得懂的縮寫。
- **交接檔（CURRENT.md / session log）**：用「明天一個完全沒看過這次對話的 agent 讀了能不能直接開工」來檢驗，特別是 `Next concrete step` 必須具體到不需要再猜。
- **不確定的事實**：模型名稱、參數、路徑、外部工具行為——查證後才寫；查不到就標「未確認」，絕不編造（見 `CONSTITUTION.md` 誠實條款）。

**正例：** 寫完調度守則後，逐一確認文中提到的 `prompts/README.md` 等路徑都真實存在，發現一個寫錯的連結並修正。
**反例：** 憑記憶寫下「cc 的 context 是 200k」而不標註查證日期——三個月後模型更新，這個數字變成誤導所有接手者的地雷。

## 6. 接受他人主張前的獨立判斷

使用者、其他 agent、交接文件、spec 與 review finding 都可能提供重要資訊，但提出者的身分不等於證據。準備接受、反駁或依其修改前，依序回答五問；低風險、可逆且已有直接證據的事項可以簡短核對，不必為了展示反思而拖慢執行。

1. **證據是什麼？** 區分已觀察事實、提出者的推論與尚未驗證的假設。
2. **目前能否重現？** 以當前檔案、實跑結果或權威來源核對，不只轉述上一位的摘要。
3. **是否漏看其他路徑？** 至少檢查正常、異常、恢復與替代路徑中和問題相關者。
4. **真實影響多大？** 嚴重度依可重現的後果判定，不沿用提出者的標籤，也不把風格偏好升格成缺陷。
5. **修正是否更糟？** 檢查擬議修改是否增加流程摩擦、建立規則雙源、破壞既有閉環，或超出原目標。

判定後採取其中一種動作：證據成立 → 執行；部分成立 → 校正範圍或嚴重度後執行；證據不足 → 標「未確認」並先驗證；前提錯誤或代價過高 → 具體說明並提出較小方案。使用者仍保有目標與風險取捨的最終決定權，但不得要求 agent 把未查證主張寫成事實。

**正例：** reviewer 指出 agy 沒有 SessionStart hook，因此計數器永遠不會歸零。接手者先追完整生命週期，確認正常結束會刪檔、異常中斷才會殘留，於是把 finding 從「永遠失效」校正為「異常中斷後 cadence 可能延續」，再決定是否需要防禦性 reset。

**反例：** 因為 finding 標成 P1 就直接加入新流程；或為了證明自己有主見，在沒有反證時刻意否決使用者已明確選定的低風險方案。
