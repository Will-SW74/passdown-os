# 模型調度守則 (Dispatch Rules)

本檔管的是「**一個 session 內部**」怎麼調度模型與 subagent；`CONSTITUTION.md` 管的是「session **之間**」怎麼交接。兩者互補。

適用範圍：
- 支援 subagent 的工具（例如 cc 的 subagent 機制）：全文適用。cc 環境已把本守則自動化成現成的 subagent 範本組，見 [`entrypoints/claude-agents/`](entrypoints/claude-agents/README.md)。
- 不支援 subagent 的工具：第 1 節跳過，第 2～6 節同樣適用於「把任務交給另一個 agent 或下一個 session」的情境。

## 1. 指揮官不下場

主對話（通常是較貴的模型）只做「換便宜模型就會掉品質」的事：任務拆解、方案決策、審查結論、寫交接。以下工作一律交辦給便宜的 subagent，主對話只收結論：

- 大量讀檔、掃 repo、找出「某個東西在哪裡」
- 網頁搜尋與資料彙整
- 批次改檔（模式已確定，只是重複套用）
- 跑測試、跑驗證、read-back 檢查
- 格式化、整理、產生樣板內容

**正例：** 要找出專案中所有呼叫 `parseHeader` 的位置 → 派便宜 subagent 搜尋，回報「檔案:行號」清單，主對話再決定怎麼改。
**反例：** 主對話自己逐個開檔案找呼叫點，燒掉一半 context 之後才開始做真正需要判斷的事。

## 2. 任務交辦三要素

每次交辦（給 subagent、給另一個 agent、或寫進交接）必含三要素，缺一不可：

1. **目標與動機** — 不只說「做什麼」，也說「為什麼」，讓執行者在遇到邊界情況時能自行做出符合意圖的判斷。
2. **驗收條件** — 可檢查的完成定義，逐條列出。「處理好 XX」不是驗收條件；「`npm test` 全綠且新增至少一個涵蓋空字串輸入的測試」才是。
3. **回報格式** — 明確說要回什麼、用什麼格式（見第 4 節回報合約）。

填空範本見 [`prompts/`](prompts/README.md)，依任務型態（搜尋／實作／重構／研究／審查）選用。

## 3. 顯式指定 model 與 effort

交辦時明確指定 model 與 effort（該工具支援時）。原則：

- **機械式大量作業**（搜尋、批次套用、格式整理）→ 當下環境最便宜的可用模型，低 effort。
- **一般實作**（有明確 spec 的 coding）→ 中階模型，中 effort。
- **需要判斷的工作**（審查、方案比較、除錯策略）→ 高階模型，或用多樣本評審（見第 6 節）。

**模型名稱與參數絕不憑記憶填。** 先查當下環境實際可用的清單；查不到就用該工具預設值，並在回報中標「model/effort 未指定，使用預設」。

## 4. 回報合約

subagent（或被交辦的 agent）回報時只回：

1. **結論** — 一段話，直接回答交辦的問題。
2. **證據** — `檔案路徑:行號` 形式的具體位置；長產物（報告、大段程式碼）寫進檔案，回傳路徑，不在回報中貼全文。
3. **完成聲明** — 明確說「驗收條件 X 條中 Y 條通過」；有沒做到的就列出來，不可用「大致完成」「應該可以」帶過。

禁止：回傳整個檔案內容、回傳未經驗證的猜測、把「我改了」當成「改對了」。

## 5. 升降級路徑（具體數字，不憑感覺）

- **便宜模型做錯 1 次** → 換中階模型重派，交辦時附上第一次的錯誤內容。
- **中階模型在同一子任務連錯 2 次** → 帶完整失敗軌跡（試了什麼、錯在哪、錯誤訊息）升級到高階模型，或停下來問使用者。
- **高階模型解出「模式」後** → 把解法寫成可照抄的步驟，降回便宜模型批次套用到其餘同類位置。
- **同一件事不論誰做，最多重試兩輪**。仍失敗 → 依 `CONSTITUTION.md` 卡關規則寫進 Blockers，停止。
- 重試前先過一遍 [`RUBRICS.md`](RUBRICS.md) 的「換路訊號」——有時該換的不是模型，是方向。

## 6. 驗證不自驗

做的人不驗收自己的產出。驗證一律派 **fresh-context**（乾淨上下文、沒看過實作過程）的 subagent，或留給下一個 session：

- **檔案產出** → read-back：重新讀檔，確認檔案存在、內容完整、沒有 `<佔位符>` 殘留。
- **程式碼** → 跑測試或實際執行，以輸出為準；不接受「看起來對」。
- **高風險判斷**（架構決策、破壞性變更）→ 第二意見：另派一個 agent 用不同路徑做一次，或產生多個候選答案後評審擇優。

為什麼要 fresh-context：做的人會帶著「我剛才是這樣想的」的偏見去驗，看不到自己的盲點；乾淨上下文的驗證者只看得到產出本身，這才是下一個接手者真實的視角。

## 7. 跨 agent 調度（codex / cc / agy 怎麼選）

通用原則：

- **要一次裝進很多檔案的工作**（大範圍讀碼、跨子系統重構評估、長文件研究）→ 派 CT 大的 agent。
- **精細、多輪工具呼叫的實作** → 派你實測過最穩的 coding agent。
- 各工具的實際 CT、可用模型、強項**因版本與訂閱方案而異，以當下查證與實際體感為準**，不可沿用舊值。
- cc 環境的「任務型態 → subagent → model」自動路由已做成範本組（`entrypoints/claude-agents/`），搭配 CLAUDE.md 的「調度模式」段落使用。
